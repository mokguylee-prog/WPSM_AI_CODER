기반 모델: qwen2.5-coder-7b-instruct-q4_k_m.gguf
목표: 로컬/경량 모델로 Claude Code 비슷한 코딩 에이전트 만들기
핵심 과제: 모델 성능보다도 도구 연결, 프롬프트 구조, 파일 편집 흐름, 컨텍스트 관리가 더 중요함

솔직히 말하면, 7B GGUF 양자화 모델만으로 Claude Code 수준의 “추론 + 장기 컨텍스트 + 안정적 코드 수정”을 그대로 재현하는 건 어렵습니다.
대신 **“작업 범위를 좁힌 로컬 코드 에이전트”**로 만들면 꽤 쓸 만하게 나옵니다.

핵심은 이겁니다.

1. 모델을 “채팅 AI”가 아니라 “도구를 쓰는 실행기”로 써야 함

Claude Code처럼 보이게 하려면 모델이 직접 모든 걸 해결하는 게 아니라:

파일 읽기
grep/search
코드 수정
테스트 실행
git diff 확인
오류 로그 요약

이런 걸 툴 호출 방식으로 연결해야 합니다.

즉:

LLM = 판단/계획
외부 프로그램 = 실제 작업

이 구조가 되어야 합니다.

2. 작은 모델일수록 한 번에 큰 작업을 맡기면 망가짐

qwen2.5-coder-7b-instruct-q4_k_m.gguf는 가볍고 좋지만:

큰 리팩토링
여러 파일 동시 수정
깊은 추론
긴 오류 로그 해석
복잡한 아키텍처 설계

에서는 흔들릴 가능성이 큽니다.

그래서 작업 단위를 이렇게 쪼개는 게 좋습니다:

1단계: 문제 이해
2단계: 관련 파일 찾기
3단계: 수정 후보 제안
4단계: 패치 생성
5단계: 테스트 실행
6단계: 실패 시 재수정

이런 반복 루프형 에이전트가 잘 맞습니다.

3. Claude Code 느낌은 “모델”보다 “워크플로우”에서 남

비슷하게 만들려면 아래 4개가 중요합니다.

A. 시스템 프롬프트

모델에게 역할을 강하게 고정합니다.

예:

너는 코드베이스 수정 에이전트다
모르면 추측하지 말고 파일을 먼저 읽어라
수정 전 관련 파일을 탐색해라
항상 변경 이유를 짧게 설명해라
한 번에 너무 많이 바꾸지 마라
테스트 가능한 경우 먼저 테스트 명령을 제안/실행해라
B. 도구 세트

최소한 이것들은 있어야 합니다.

read_file(path)
write_file(path, content)
search_code(query)
list_files(path)
run_command(cmd)
apply_patch(diff)
show_diff()
C. 컨텍스트 압축

작은 모델은 긴 대화를 다 넣으면 금방 망가집니다.
그래서 매 턴마다 전체 로그를 넣지 말고:

현재 목표
관련 파일 요약
최근 수정 내용
테스트 결과
다음 행동

이 5개 정도만 메모리 요약으로 유지하는 게 좋습니다.

D. 패치 우선

작은 모델은 파일 전체를 다시 쓰게 하면 사고가 납니다.
가능하면:

전체 파일 재생성보다
부분 패치(diff/apply_patch)

방식이 훨씬 안정적입니다.

4. 이 모델로 기대할 수 있는 현실적인 수준

잘 만들면:

단일 프로젝트에서
파일 몇 개 범위 내 수정
버그 수정 보조
함수 추가
로그 분석
테스트 실패 원인 추정
반복적인 코드 편집

정도는 꽤 괜찮을 수 있습니다.

하지만 약한 부분은:

긴 추론
복잡한 멀티파일 리팩토링
애매한 요구사항 해석
한 번에 완성도 높은 대규모 변경

입니다.

5. 실전 구조는 이런 식이 좋습니다

추천 구조:

사용자 입력 수신
모델이 해야 할 일 분해
관련 파일 탐색
필요 파일만 모델에 주입
패치 생성
테스트/린트 실행
결과를 다시 모델에 넣고 수정
최종 요약 + diff 출력

즉, “한 번에 답변”이 아니라
탐색 → 수정 → 검증 루프로 가야 합니다.

6. 이 모델로 만들 때 특히 주의할 점

q4_k_m.gguf는 가볍지만 정확도 손실이 있으니:

추천 구조

Claude Code 비슷하게 만들려면 구조를 이렇게 잡는 게 좋습니다.

1. 모델은 두뇌, 실제 작업은 툴

모델이 직접 다 하지 말고 아래 툴을 붙입니다.

list_files
read_file
search_code
apply_patch
run_command
show_diff

역할 분리:

LLM: 무엇을 할지 판단
툴: 실제 파일 읽기/수정/테스트

이 구조가 핵심입니다.

2. 한 번에 큰 작업 금지

작은 모델은 한 번에 크게 시키면 헛소리 확률이 올라갑니다.

좋은 흐름:

문제 이해
관련 파일 찾기
읽을 파일 최소화
수정안 제안
패치 적용
테스트 실행
실패하면 재수정

이렇게 루프형으로 돌아야 합니다.

3. 프롬프트를 강하게 고정

시스템 프롬프트가 꽤 중요합니다.

예시 방향:

너는 코드 수정 에이전트다
추측하지 말고 먼저 파일을 읽어라
관련 파일을 찾은 뒤 수정하라
전체 파일 재작성보다 patch를 선호하라
명령 실행 전 위험 여부를 점검하라
변경 후 diff와 이유를 짧게 요약하라

작은 모델일수록 이런 규칙이 효과가 큽니다.

4. 컨텍스트는 요약해서 유지

Claude Code처럼 긴 대화를 다 넣으면 7B 모델은 금방 흔들립니다.

매 턴 유지할 정보는 이것만 압축해도 됩니다:

현재 목표
관련 파일 목록
최근 변경 사항
최근 테스트 결과
다음 액션

즉, 전체 대화 기억보다 작업 상태 메모가 중요합니다.

가장 중요한 팁

이 모델로 만들 때는 파일 전체 재작성보다 diff/patch 방식이 훨씬 낫습니다.

왜냐하면:

작은 모델은 기존 코드 보존을 자주 실패함
양자화 모델은 사소한 누락이 더 생김
patch는 변경 범위를 강제로 줄여줌

그래서 출력 포맷도 가능하면 이런 식이 좋습니다:

행동: read/search/patch/run
대상 파일
변경 이유
패치 내용
구현 레벨에서 추천
최소 기능 v1

처음엔 이것만 있어도 됩니다.

파일 검색
파일 읽기
부분 수정

결과 요약
