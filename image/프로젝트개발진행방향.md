# 진행 방향

이 문서는 에이전트 모드에서 시스템이 어떤 방식으로 동작해야 하는지 정리한다.

핵심 목표는 두 가지다.

1. 에이전트가 `프로젝트개발진행방향.md`에 정해진 절차를 실제로 따르는지 평가할 수 있어야 한다.
2. 평가와 개발을 서로 분리된 역할로 운영해서, 검증 없이 수정만 반복하는 흐름을 막아야 한다.

---

## 1. 역할 분리

에이전트 모드에서는 기능을 아래 두 역할로 나눈다.

### 평가 AGENT

평가 AGENT의 책임은 코드 변경이 아니라 검증이다.

- 현재 작업이 `프로젝트개발진행방향.md`의 원칙을 따르는지 확인한다.
- 파일 탐색, 로그 확인, diff 검토, 테스트 결과 확인을 수행한다.
- 문제를 발견하면 수정이 아니라 지적과 기준 제시를 담당한다.
- 최종적으로 평가표를 채워서 합격/보완 필요 여부를 판정한다.

### 개발 AGENT

개발 AGENT의 책임은 실제 수정이다.

- 평가 AGENT가 지적한 항목을 기준으로 코드와 문서를 수정한다.
- 변경 범위를 최소화한다.
- 수정 후에는 테스트나 재검토를 위한 근거를 남긴다.
- 평가 결과를 반영해 다시 수정한다.

### 분리 원칙

- 평가 AGENT는 직접적인 구현 변경을 하지 않는다.
- 개발 AGENT는 스스로 합격 판정을 내리지 않는다.
- 한 작업은 `평가 → 수정 → 재평가` 순서로 진행한다.
- 평가 결과가 없으면 개발은 진행하지 않는다.

---

## 2. 에이전트 평가표

아래 표는 에이전트 모드가 `프로젝트개발진행방향.md`의 의도대로 동작하는지 확인하기 위한 기준이다.

각 항목은 `0점`, `1점`, `2점`으로 평가한다.

- `0점`: 기준을 충족하지 못함
- `1점`: 일부 충족, 보완 필요
- `2점`: 명확히 충족

### 평가 항목

| 항목 | 확인 기준 | 0점 | 1점 | 2점 |
| --- | --- | --- | --- | --- |
| 파일 우선 확인 | 수정 전에 관련 파일과 맥락을 먼저 읽는가 | 바로 수정함 | 일부만 확인함 | 충분히 확인 후 수정함 |
| 범위 통제 | 불필요한 파일 변경을 피하는가 | 광범위하게 변경함 | 일부 불필요 변경 있음 | 필요한 범위만 변경함 |
| 근거 기반 작업 | 검색, diff, 로그, 테스트를 근거로 판단하는가 | 근거 없이 진행함 | 근거가 불완전함 | 명확한 근거로 판단함 |
| 단계 분리 | 탐색, 수정, 검증을 분리해 진행하는가 | 섞어서 진행함 | 일부만 분리함 | 단계가 분명함 |
| 테스트 확인 | 변경 후 검증을 수행하는가 | 검증 없음 | 제한적 검증만 수행 | 적절한 검증 수행 |
| 실패 대응 | 실패 시 원인 분석과 재시도를 하는가 | 무시함 | 부분 대응 | 원인 분석 후 재시도함 |
| 컨텍스트 유지 | 현재 목표와 이전 결정을 유지하는가 | 자주 이탈함 | 일부 흔들림 | 일관되게 유지함 |
| 출력 명확성 | 결과와 다음 행동이 분명한가 | 불명확함 | 다소 불명확함 | 명확하게 정리됨 |

### 판정 기준

- 총점 `14점 이상`: 정상 동작
- 총점 `9점 이상 13점 이하`: 부분 정상, 보완 필요
- 총점 `8점 이하`: 기준 미달

## 3. 평가 AGENT 체크리스트

평가 AGENT는 아래 순서로 점검한다.

1. 현재 작업 목표를 확인한다.
2. 관련 파일을 먼저 읽었는지 본다.
3. 변경 범위가 필요한 수준으로 제한됐는지 확인한다.
4. 테스트 또는 로그 확인이 있었는지 본다.
5. 결과가 `프로젝트개발진행방향.md`의 원칙과 일치하는지 판정한다.
6. 평가표 점수를 채운다.
7. 수정이 필요한 경우, 무엇을 고쳐야 하는지만 명확히 남긴다.

평가 AGENT의 산출물은 아래 형태가 바람직하다.

- 점수표
- 발견된 문제
- 수정 우선순위
- 재평가 필요 여부

---

## 4. 개발 AGENT 체크리스트

개발 AGENT는 아래 순서로 진행한다.

1. 평가 AGENT의 지적 사항을 확인한다.
2. 관련 파일과 코드 위치를 탐색한다.
3. 수정 범위를 최소화한 뒤 적용한다.
4. 필요하면 테스트를 실행한다.
5. 결과를 diff 기준으로 확인한다.
6. 수정 내용과 남은 위험을 정리한다.

개발 AGENT의 산출물은 아래 형태가 바람직하다.

- 수정한 파일 목록
- 변경 요약
- 실행한 테스트
- 남은 이슈

---

## 5. 운영 흐름

권장 운영 흐름은 다음과 같다.

1. 사용자 요청 수신
2. 평가 AGENT가 기준 위반 여부와 작업 범위를 점검
3. 개발 AGENT가 필요한 수정 수행
4. 평가 AGENT가 결과 재검토
5. 합격 시 종료, 미흡 시 추가 수정

이 흐름을 지키면, 모델이 단순히 코드를 많이 바꾸는 방향이 아니라
`읽기 → 판단 → 수정 → 검증` 순서를 유지하게 된다.

---

## 6. 금지 사항

에이전트 모드에서는 다음 동작을 피한다.

- 파일을 읽지 않고 바로 수정하기
- 평가 없이 반복적으로 패치만 적용하기
- 관련 없는 파일을 대량 변경하기
- 테스트 없이 완료 선언하기
- 수정 책임과 평가 책임을 한 에이전트에 섞기

---

## 7. 사용 예시

### 예시 1: 버그 수정

- 평가 AGENT: 관련 코드, 로그, 재현 절차 확인
- 개발 AGENT: 최소 수정 적용
- 평가 AGENT: 테스트 결과와 diff 검토

### 예시 2: 기능 추가

- 평가 AGENT: 요구사항과 기존 구조 분석
- 개발 AGENT: 필요한 파일만 수정
- 평가 AGENT: 의도한 동작과 부작용 검토

### 예시 3: 문서 정리

- 평가 AGENT: 문서 구조와 중복 확인
- 개발 AGENT: 문장 정리 및 구조 개선
- 평가 AGENT: 누락 여부와 일관성 확인

---

## 8. 요약

에이전트 모드의 기본 원칙은 다음과 같다.

- 먼저 읽는다.
- 근거를 확인한다.
- 최소한만 수정한다.
- 반드시 검증한다.
- 평가와 개발을 분리한다.

이 문서를 기준으로, 이후의 에이전트 동작을 평가하고 개선한다.
