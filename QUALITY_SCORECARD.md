# Quality Scorecard

작성일: 2026-04-18

## Goal

이 프로젝트의 기본 목표는 다음 원칙을 만족하는 것이다.

- 서버는 문제가 생겨도 스스로 복구 가능한 구조여야 한다.
- 클라이언트는 입력의 성공/실패를 명확히 알려야 한다.
- 실패하면 원인을 분류해서 사용자에게 전달해야 한다.
- 한 세션의 실패가 전체 서버 장애처럼 보이면 안 된다.
- 애매한 상태를 만들지 않는다.

## Scoring

총점은 100점이며, 95점 이상이어야 합격으로 본다.

### 1. Server resilience - 30 points

- [ ] 10점: 서버가 단일 세션 실패로 죽지 않는다.
- [ ] 10점: 자동 재시도, timeout, health check, cleanup이 있다.
- [ ] 10점: 세션/작업/연결 상태가 분리되어 있다.

### 2. Client feedback clarity - 20 points

- [ ] 10점: 성공, 실패, 진행 중이 분명히 구분된다.
- [ ] 10점: 실패 원인이 사용자에게 바로 보인다.

### 3. Reconnect stability - 20 points

- [ ] 10점: 연결을 끊고 다시 붙어도 세션이 꼬이지 않는다.
- [ ] 10점: 재연결 후 새 작업을 보내도 서버가 offline으로 오인되지 않는다.

### 4. Observability - 15 points

- [ ] 5점: health endpoint가 안정적으로 동작한다.
- [ ] 5점: stats나 보조 정보 실패가 서버 다운으로 오인되지 않는다.
- [ ] 5점: 로그가 원인 파악에 충분하다.

### 5. Recovery workflow - 15 points

- [ ] 5점: 실패 시 다음 행동이 안내된다.
- [ ] 5점: 자동 복구와 수동 복구의 경계가 분명하다.
- [ ] 5점: 재현 가능한 실패 케이스를 기준으로 수정할 수 있다.

## Acceptance Rule

- 95점 이상: 합격
- 90~94점: 개선 반복
- 89점 이하: 구조 수정 우선

## Required Loop

1. 현재 상태를 점수화한다.
2. 95점 미만 항목을 우선 수정한다.
3. 수정 후 다시 점수화한다.
4. 95점 이상이 될 때까지 반복한다.

## Current Direction

이번 저장소에서는 특히 아래를 우선 개선한다.

- 재연결 안정성
- 서버/세션/작업 상태 분리
- 실패 원인 전달
- 보조 API 실패가 전체 offline으로 번지지 않도록 분리
