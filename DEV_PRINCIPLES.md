# Development Principles

작성일: 2026-04-18

## Core Rule

서버는 문제가 생겨도 스스로 복구 가능한 구조여야 하고, 클라이언트는 입력과 출력의 성공/실패를 명확히 알려주며 실패 원인까지 사용자에게 전달해야 한다.

## Server Principles

- 서버는 가능하면 죽지 않아야 한다.
- 장애가 나면 전체 종료보다 세션/작업 단위로 격리한다.
- 연결 문제, 작업 문제, 세션 문제를 서로 분리해서 처리한다.
- 자동 재시작, 타임아웃, 헬스체크, 상태 복구 경로를 둔다.
- 오래된 세션이나 비정상 연결은 서버 전체가 아니라 해당 단위만 정리한다.

## Client Principles

- 사용자의 입력은 반드시 성공, 실패, 진행 중 상태로 구분해서 보여준다.
- 실패하면 단순 실패가 아니라 원인까지 알려준다.
- 실패 원인은 최소한 다음 범주로 나눈다.
  - 연결 실패
  - 서버 오류
  - 세션 오류
  - 작업 오류
  - 권한/승인 실패
- 실패 시 사용자가 수정하거나 개선할 수 있도록 다음 행동을 제안한다.

## Workflow Principles

- 성공은 성공답게 끝내고, 실패는 숨기지 말고 드러낸다.
- 재시도는 무작정 하지 말고 원인을 먼저 파악한다.
- 로그와 상태는 항상 남긴다.
- 애매한 상태를 만들지 않는다.
- 서버와 클라이언트는 서로의 책임을 혼동하지 않는다.

## Application Rule

이 원칙은 이 저장소의 모든 프로젝트와 기능에 공통 적용한다.
